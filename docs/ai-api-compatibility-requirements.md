# 多平台 AI API 兼容层需求文档

## 1. 文档目的

本文档用于指导项目实现统一的多平台 AI API 兼容层，解决不同 OpenAI-compatible 平台在请求格式、流式返回、参数支持和错误处理上的差异问题。

目标是让开发同事可以直接按本文档完成设计、编码、测试和验收，不需要额外口头补充需求。

## 2. 背景问题

当前项目在 `src/baodou_ai/ai/client.py` 中直接面向不同平台发送 OpenAI-compatible 请求。业务逻辑本身可以正常工作，但不同平台在兼容细节上存在明显差异，常见问题包括：

- 纯文本消息既有字符串形式，也有 `content part` 数组形式，不同平台支持程度不同。
- 某些平台不接受空文本 `part`，例如 `{"type": "text", "text": ""}`，甚至会将其识别为缺少 `text` 字段。
- 某些平台不支持 `stream_options.include_usage`。
- 某些平台不支持 `reasoning_effort`、`thinking` 或其他扩展字段。
- 某些平台对多模态消息、图片 data URL、工具调用参数的校验更严格。
- 不同平台的错误码、错误文案、usage 返回结构、流式 chunk 格式存在差异。

当前问题已经导致项目在切换不同平台时，需要临时增加平台特判，维护成本高，且容易引入回归问题。

## 3. 需求目标

本次需求的目标如下：

- 建立统一的 AI API 兼容层，屏蔽不同平台之间的请求和响应差异。
- 保持现有业务逻辑基本不变，不修改核心控制循环行为。
- 让 `runner`、`memory`、`prompt_builder` 等业务模块继续使用项目内部统一的数据结构。
- 将平台差异集中收敛到兼容层中，避免兼容逻辑散落在业务代码中。
- 支持后续低成本新增平台，而不是继续复制平台特判代码。
- 建立稳定的测试矩阵，防止兼容性修复后影响其他平台。

## 4. 不在本次范围内

以下内容不属于本次需求范围：

- 不重写 `ControlLoopRunner` 的控制循环逻辑。
- 不重写 `MemoryManager` 的业务策略，例如是否保留历史消息、是否裁剪图片。
- 不改动现有 prompt 业务语义。
- 不改动模型输出协议本身。
- 不引入新的第三方模型 SDK，仍以现有 OpenAI SDK 为主。
- 不承诺所有平台功能完全一致，只要求兼容层能做能力识别、自动降级和稳定回退。

## 5. 总体设计要求

### 5.1 设计原则

- 业务层只面向项目内部统一协议，不直接面向平台差异。
- 所有平台兼容逻辑必须集中在兼容层，不允许散落在 `runner`、`memory`、`prompt_builder` 等业务模块中。
- 兼容层必须支持请求规范化、能力裁剪、响应归一化、错误识别和自动回退。
- 必须优先采用“发送前规范化”的方式解决兼容问题，而不是修改业务层原始数据。
- 新增平台时，应当主要通过新增 profile 或 adapter 完成，不应修改大量现有业务代码。

### 5.2 架构要求

需要在 `src/baodou_ai/ai/` 下新增兼容层相关模块，至少包含以下职责：

- 平台识别
- 平台能力描述
- 请求规范化
- 请求裁剪和降级
- 响应归一化
- 错误识别和回退重试

建议结构如下：

```text
src/baodou_ai/ai/
├── client.py
├── provider_profiles.py
├── provider_adapter.py
├── request_normalizer.py
├── response_normalizer.py
└── retry_policy.py
```

说明：

- 文件名可根据现有代码风格调整。
- 允许合并部分模块，但职责边界必须保留。
- 不允许继续把所有兼容逻辑堆叠在 `client.py` 一个文件中。

## 6. 内部统一协议要求

兼容层必须先定义项目内部统一请求模型。业务层继续产出内部模型，再由兼容层映射到目标平台请求体。

### 6.1 统一请求模型要求

统一请求模型至少需要覆盖以下字段：

- `model`
- `messages`
- `stream`
- `extra_body`
- `stream_options`
- `tools`
- `tool_choice`
- `temperature`
- `response_format`

### 6.2 `messages` 内部标准

内部消息允许存在以下形式：

- `content` 为普通字符串
- `content` 为数组，数组元素可能为：
  - 文本 part
  - 图片 part

内部消息的允许范围可以相对宽松，但在发送前必须经过规范化。

### 6.3 发送前规范化要求

在真正调用平台 API 前，必须对消息进行统一清洗，至少包含以下规则：

- `type == "text"` 的 part 必须补齐 `text` 字段。
- `text is None` 时必须转为字符串。
- 空字符串文本 part 必须根据平台能力决定删除、保留或降级。
- `type == "image_url"` 的 part 必须保证存在 `image_url.url`。
- 非法 part 必须在发送前剔除。
- 空的 `content` 数组不能直接发送。
- 纯文本消息优先根据平台要求降级为字符串 `content`。
- 清洗过程不能修改原始内存对象，必须使用副本。

## 7. 平台能力模型要求

兼容层必须引入平台能力描述对象，用于描述不同平台支持什么、不支持什么，以及应该如何降级。

每个平台至少要支持以下能力声明：

- 是否支持纯文本字符串消息
- 是否支持纯文本 `content part` 数组
- 是否支持空文本 part
- 是否支持多模态图片消息
- 是否支持图片 data URL
- 是否支持 `stream_options.include_usage`
- 是否支持 `reasoning_effort`
- 是否支持 `thinking`
- 是否支持工具调用
- 是否支持并行工具调用
- 是否支持特定 `response_format`

要求：

- 平台能力必须集中定义，不能在请求发送代码中到处写 `if "xxx" in base_url`。
- `base_url` 到平台类型的映射逻辑必须统一管理。
- 必须为未知平台提供默认 profile，默认策略应以保守兼容为主。

## 8. 请求适配要求

兼容层必须在发送前，根据目标平台 profile 对内部请求做适配。

### 8.1 适配范围

至少需要适配以下内容：

- `messages`
- `extra_body`
- `stream_options`
- 工具调用相关字段
- 响应格式字段

### 8.2 适配策略

请求适配必须遵循以下规则：

- 平台不支持的字段必须在发送前剔除。
- 多种兼容写法同时存在时，优先选择平台兼容性最高的写法。
- 对文本消息，优先使用最简单、最保守的可接受格式。
- 当平台不支持某项高级能力时，必须自动降级，而不是直接报错结束。

示例要求：

- 某平台不支持 `stream_options.include_usage` 时，应自动去掉该字段。
- 某平台不支持 `reasoning_effort` 时，应自动去掉该字段。
- 某平台不支持文本 `content part` 数组时，应将纯文本消息降级为普通字符串。
- 某平台不支持空文本 part 时，应删除空文本 part；如果整条消息因此为空，则应跳过该消息或替换为可接受的占位内容。

## 9. 响应适配要求

兼容层必须对不同平台的响应做归一化，供现有业务层继续消费。

### 9.1 非流式响应

必须统一输出以下能力：

- 主文本内容提取
- usage 提取
- reasoning token 提取
- 工具调用提取
- 缺失字段兜底

### 9.2 流式响应

必须统一处理以下场景：

- 正常文本 chunk
- 空 chunk
- usage 在最后一个 chunk 返回
- usage 完全不返回
- 平台扩展字段存在或缺失

要求：

- 业务层不得感知不同平台的 chunk 差异。
- 流式响应解析失败时，必须给出明确日志和可定位信息。

## 10. 自动回退要求

兼容层必须具备自动回退能力，避免因为单个平台的局部不兼容直接终止任务。

### 10.1 回退触发条件

以下场景必须触发自动回退：

- 请求参数不被识别
- 扩展字段不被支持
- 流式 usage 参数不被支持
- 消息结构校验失败

### 10.2 回退顺序要求

建议至少支持如下顺序的回退策略：

1. 去掉 `stream_options`
2. 去掉 `reasoning_effort`
3. 去掉 `thinking` 或其他扩展字段
4. 将纯文本 part 数组降级为字符串消息
5. 删除空文本 part 和非法 part
6. 将多模态消息降级为更保守格式

要求：

- 回退必须有明确顺序，不能无序叠加特判。
- 每次回退必须记录日志，说明触发原因和本次降级动作。
- 相同平台的失败能力可以缓存，避免同一轮任务反复试错。

## 11. 日志与调试要求

兼容层必须增强调试信息，便于定位平台兼容问题。

至少需要记录以下信息：

- 当前识别到的平台类型
- 当前使用的平台 profile
- 发送前的规范化动作
- 被删除或降级的字段列表
- 自动回退的触发原因
- 回退前后的关键请求差异
- 响应归一化异常信息

要求：

- 日志中不能输出 API Key。
- 图片 data URL 不能完整写入日志，可只保留长度或摘要。
- 调试日志必须能支持复现兼容问题。

## 12. 对现有代码的改造边界

### 12.1 必须改动的模块

- `src/baodou_ai/ai/client.py`
- 新增的兼容层模块
- 对应的测试文件

### 12.2 尽量不改动的模块

- `src/baodou_ai/core/runner.py`
- `src/baodou_ai/ai/memory.py`
- `src/baodou_ai/ai/prompt_builder.py`
- `src/baodou_ai/agent/`

### 12.3 禁止事项

- 禁止为某个平台在业务层直接增加散乱特判。
- 禁止为了兼容某个平台而改变现有业务 prompt 语义。
- 禁止把“平台不支持”直接当作“业务异常”传播到上层。
- 禁止通过修改内存中的原始消息对象来规避兼容问题。

## 13. 测试要求

必须补齐兼容层测试，不能只依赖手工联调。

### 13.1 单元测试

至少覆盖以下场景：

- 文本消息字符串输入
- 文本消息 `content part` 输入
- 空文本 part 清洗
- 非法 part 清洗
- 图片消息合法化
- `stream_options` 剔除逻辑
- `reasoning_effort` 剔除逻辑
- 平台 profile 识别
- 自动回退顺序

### 13.2 回归测试

必须新增或更新以下类型的测试：

- 硅基流动兼容用例
- 火山或其他已知平台兼容用例
- 默认未知平台保守兼容用例
- 多轮对话历史裁剪后的请求兼容用例

### 13.3 测试约束

- 测试应尽量基于 mock client，不依赖真实外网服务。
- 对平台差异应通过伪造响应和伪造错误完成验证。
- 必须验证“回退后可成功继续执行”这一行为，而不是只验证“发生了异常”。

## 14. 验收标准

以下条件全部满足，方可视为本需求完成：

- 新兼容层已落地，平台差异逻辑集中管理。
- `client.py` 不再继续增长大量平台散乱特判。
- 现有业务执行链在主流程上保持不变。
- 已知平台的关键兼容问题可以通过适配层解决。
- 空文本 part、非法 part、平台不支持字段等问题可在发送前被清洗或降级。
- `stream_options`、`reasoning_effort` 等扩展能力可根据平台自动启用或禁用。
- 流式和非流式调用均可走统一兼容层。
- 单元测试和回归测试覆盖主要兼容场景并通过。
- 调试日志足以定位兼容失败原因。

## 15. 开发任务拆分

建议按以下顺序实施：

1. 定义平台 profile 和平台识别逻辑。
2. 定义统一请求模型和发送前规范化逻辑。
3. 实现请求适配器，接入 `client.py` 的流式和非流式路径。
4. 实现响应归一化逻辑。
5. 实现自动回退策略和失败能力缓存。
6. 增加日志和调试输出。
7. 完成单元测试和回归测试。
8. 验证硅基流动及其他已知平台的真实问题是否被覆盖。

## 16. 交付物要求

最终交付物至少包括：

- 兼容层代码
- 修改后的 `client.py`
- 配套测试代码
- 必要的开发文档或模块说明

如果实现过程中需要新增配置项，必须补充配置说明，并保证默认配置对现有用户行为无破坏。

## 17. 成功标准

本需求完成后的目标状态是：

- 项目内部继续只维护一套业务逻辑。
- 接入新平台时，主要工作变为补充 profile、adapter 和测试。
- 兼容问题优先通过适配层解决，而不是反复修改业务模块。
- 已知平台差异可以稳定复现、稳定定位、稳定回归验证。
