# 同窗 2.0 项目完整说明

## 1. 文档目标

这份文档面向三类人：

- 新开发者：不看代码，也能先理解项目全貌，知道从哪里开始读、改、测。
- 维护者：快速定位功能归属、配置入口、测试落点和排障路径。
- 协作者：用一份文档对齐项目的能力边界、运行方式和开发方式。

文档目标不是代替源码，而是帮助你在开始看源码前，先建立正确的地图。

## 2. 项目一句话说明

`coview` 是一个基于视觉模型的桌面自动化 Agent。  
它通过“看屏幕 -> 理解任务 -> 调用工具 -> 继续观察”的闭环，在 macOS 和 Windows 上完成桌面操作任务。

除了桌面自动化主链路，它还扩展了：

- GUI 桌面交互壳
- CLI 和 Python API
- 语音输入、本地语音唤醒与 TTS 播报
- 后台 Code Agent 子系统
- 统一日志与结构化错误治理

## 2.1 当前代码状态（2026-05）

当前项目已经完成一轮 P0 级职责收敛，阅读代码时建议带着下面这几个事实建立地图：

- 主执行入口仍然是 `ControlLoopRunner`，但轮次级分支执行已经拆到 `src/baodou_ai/core/runner_turns.py`。
- GUI 主控仍然是 `FloatingController`，但前台任务生命周期已主要落在 `TaskSessionController` + `FloatingTaskSessionHost`。
- 悬浮球的背景任务、语音唤醒/指示、前台轮询与 companion 协调已拆到独立协作模块。
- `CompanionController` 保留状态机职责，截图推荐 worker 与兼容 delegate 已拆出为独立文件。
- `ai/client.py` 和 `gui/floating/controller.py` 仍然是后续治理重点，但当前结构已经比重构前更清晰。

## 3. 先建立全局认识

### 3.1 这个项目真正的核心是什么

这个项目最重要的不是 GUI，也不是某个工具函数，而是统一运行内核 `ControlLoopRunner`。

可以把整个项目理解成三层：

1. 入口层
   - GUI
   - CLI
   - API
2. 运行内核
   - 截图
   - prompt 组装
   - 模型调用
   - 响应解析
   - 工具执行
   - 循环控制
3. 扩展能力层
   - 平台适配
   - 语音
   - TTS
   - 后台 Code Agent
   - GUI 状态管理

结论很重要：

- 三个入口不是三套业务逻辑。
- 它们最终都进入同一个 `ControlLoopRunner.run`。
- 所以，绝大多数行为改动，真正落点都在统一内核，而不是入口本身。

### 3.2 你应该如何理解这套系统

如果用“人”的比喻来理解：

- `screenshot.py` 是眼睛
- `AIClient` + prompt 体系是思考输入层
- `parser.py` + `agent/protocol.py` 是把“模型输出”变成“系统能执行的结构”
- `ToolExecutor` + `AutomationController` 是手脚
- `runner.py` 是大脑中的主循环
- `runner_turns.py` 是大脑里负责执行“这一轮具体走哪个分支”的子流程
- `gui/floating/*` 是用户能看到和操作的外壳
- `code_agent/*` 是外接的异步工作台

## 6. 入口层说明

### 6.1 GUI 入口

入口文件是 `src/baodou_ai/__main__.py`。

它主要做几件事：

- 创建 `QApplication`
- 初始化 `Config`
- 安装运行时日志缓冲
- 创建并启动 `FloatingController`

它不承载业务主逻辑。  
如果你在 GUI 启动时遇到问题，先看它；如果你在任务执行时遇到问题，通常不要停留在这里，要继续看 `runner.py` 和 GUI controller。

当前 GUI 入口往下的主路径大致是：

- `FloatingController`
- `TaskSessionController`
- `AIWorker`
- `ControlLoopRunner`

### 6.2 CLI 入口

入口文件是 `src/baodou_ai/cli.py`。

它做的事情很简单：

- 解析命令行参数
- 注入 API key/配置
- 调用 `CoViewAI.execute()`

CLI 的价值主要是：

- 脱离 GUI 做功能验证
- 做自动化脚本集成
- 在调试模型链路时减少 GUI 干扰

### 6.3 API 入口

入口文件是 `src/baodou_ai/api.py`。

它对外暴露：

- `CoViewAI`
- `execute_task`

它适合：

- 被其他 Python 项目集成
- 写实验脚本
- 用最小成本复现问题

## 7. 统一主执行链

### 7.1 核心结论

所有入口最终都会收敛到 `ControlLoopRunner.run`。

这是项目最重要的一条链路。  
如果你只能花时间读一个文件，优先读 `src/baodou_ai/core/runner.py`。

### 7.2 主循环做什么

可以把每一轮执行理解为下面 8 步：

1. 收集运行上下文
2. 获取截图或其他观察结果
3. 拼装 prompt 输入
4. 调用模型
5. 解析模型响应
6. 判断当前分支是工具、等待还是结束
7. 执行动作并得到反馈
8. 更新状态，进入下一轮或退出

当前轮次级分支执行已经从 `runner.py` 主循环中拆出，主要由 `src/baodou_ai/core/runner_turns.py` 承担：

- 工具分支执行
- `page_loading` 分支执行
- `respond` 分支执行

### 7.3 为什么每轮只执行一个主动作

这是项目的一个重要设计约束。

原因是桌面自动化和网页/系统界面变化高度耦合：

- 动作一旦执行，屏幕状态就可能变化
- 如果连续执行多个动作，中间状态无法被模型再次观察
- 这会让错误难以定位，也会放大失控风险

因此系统要求：

- 每轮只执行一个主动作
- 执行后必须重新观察
- 再由模型决定下一步

这个约束影响很多实现：

- 顶层协议设计
- runner 分支逻辑
- 工具分发方式
- 过程汇报策略
- 停滞判断

### 7.4 当前 runner 相关文件怎么分工

- `src/baodou_ai/core/runner.py`：任务级装配、观察、模型调用、主循环推进。
- `src/baodou_ai/core/runner_turns.py`：轮次级分支执行与迭代 payload 构建。
- `src/baodou_ai/core/runner_state.py`：运行循环状态对象。
- `src/baodou_ai/core/runner_metrics.py`：模型指标与 token 统计。
- `src/baodou_ai/core/context_window.py`：页面/文档提取上下文窗口管理。

## 8. 模型输入、协议与工具执行

### 8.1 prompt 是怎么组装的

模型输入不是简单的“任务文本 + 截图”，而是多部分拼接：

- 用户任务
- 当前截图
- 页面/文档提取内容
- 历史反馈
- 记忆信息
- 运行时策略信息
- 当前语言与平台上下文
- 后台任务摘要

这部分主要在：

- `src/baodou_ai/ai/client.py`
- `src/baodou_ai/ai/prompt_builder.py`
- `src/baodou_ai/ai/runtime_prompt_context.py`

如果你要改模型行为，常见落点是：

- prompt 文案
- prompt 组装顺序
- 是否把某些上下文传给模型
- 页面/文档读取结果如何进入上下文

### 8.2 顶层协议是什么

当前采用扁平分支协议。

模型返回的数据必须落在以下三类之一：

- 工具分支
- `page_loading`
- `respond`

并且可附加：

- `thinking`
- `report`
- `remember`

这套协议的目的不是追求“模型自由”，而是保证执行安全、可解释和可测试。

### 8.3 为什么有 `parser.py`

因为模型输出并不总是稳定、严格、完美。

`parser.py` 的职责是：

- 尽可能解析合法响应
- 修复轻微格式偏差
- 将输出收敛到项目要求的结构
- 对严重不合法响应做统一失败处理

如果你遇到“模型明明大意正确，但系统不执行”的问题，优先看：

- `src/baodou_ai/ai/parser.py`
- `src/baodou_ai/agent/protocol.py`

### 8.4 工具是如何被执行的

工具链路分三层：

1. `tool_registry.py`
   - 定义可用工具
   - 管理工具 schema
   - 约束参数格式
2. `tool_executor.py`
   - 按工具名分发执行
   - 统一成功/失败返回
3. `automation.py` + `automation_tools/*`
   - 具体执行桌面动作、页面读取、后台任务等

这意味着新增工具时，通常不是只改一个文件，而是至少看三层。

## 9. 后台 Code Agent 子系统

### 9.1 这个子系统解决什么问题

主执行链适合“当前屏幕上、一步一步、强依赖观察”的任务。  
但一些编码任务并不适合强依赖 GUI 观察，例如：

- 改仓库代码
- 跑命令
- 生成补丁
- 分析日志

所以项目单独做了一个后台 Code Agent 子系统。

### 9.2 它和主链路的关系

它不是替代 `ControlLoopRunner`，而是被主链路通过工具触发。

可以理解为：

- 主链路负责“发起后台任务”
- 后台系统负责“异步执行”
- 主链路负责“把后台状态继续反馈给模型和 UI”

### 9.3 核心组成

- `manager.py`
  - 管理任务生命周期、状态、取消、结果归档
- `runtime.py`
  - 处理任务运行期的落盘、会话信息、工件管理
- `dispatcher.py`
  - 根据 provider 分发到不同 adapter
- `adapters/*`
  - 具体封装 `codex`、`claude`、`kimi`、`qwen`、`codebuddy` 等 CLI agent
- `models.py`
  - 任务状态、结果、报告等结构定义

### 9.4 什么时候看这一层

遇到以下需求时直接进 `code_agent/`：

- 新增 provider
- 修改 CLI 命令模板
- 调整后台任务超时/并发
- 修改任务状态展示
- 调整后台结果如何汇报到 GUI 或模型

## 10. GUI 层说明

### 10.1 GUI 的职责

GUI 不是单纯的显示层，它还承担：

- 任务发起
- 任务停止
- 运行态展示
- 日志展示
- 配置编辑
- 语音交互与本地唤醒编排
- 后台任务窗口管理

### 10.2 你最应该知道的 GUI 文件

- 改 companion / 本地唤醒 / 后台轮询协作：看 `controller_delegates.py`、`background_activity_coordinator.py`

- 改配置展示：看 `control_console.py` + `config.py`

这是 GUI 开发最容易混淆的地方。

## 11. 语音、本地唤醒与 TTS

### 11.1 语音链路做什么

语音子系统主要覆盖：

- 采集麦克风音频
- VAD 判断是否有有效语音
- ASR 识别文本
- 意图分类
- 决定是停止任务、发起新任务还是忽略

### 11.2 本地语音唤醒做什么

本地语音唤醒是这次新增的一条轻量常驻链路，目标是在悬浮球未固定、且当前没有活跃任务时，允许用户直接说唤醒词进入现有语音交互。

当前设计要点：

- 唤醒词识别完全本地进行，使用 `sherpa-onnx`
- 默认内置两个唤醒词：`你好小彤` / `hey Lucy`
- 用户可以在控制台的语音设置页修改中文和英文唤醒词
- 唤醒成功后自动激活悬浮球，并复用现有语音交互链路
- 唤醒命中后会立即通过 TTS 播报“我在”，给用户声音反馈
- 正在完整语音交互、任务执行或悬浮球固定时，wake-word 监听不会抢占主流程

核心落点：

- `src/baodou_ai/gui/floating/controller.py`
- `src/baodou_ai/gui/floating/controller_delegates.py`
- `src/baodou_ai/gui/control_console.py`
- `src/baodou_ai/voice/sherpa_keyword_spotter.py`
- `src/baodou_ai/voice/wake_word_engine.py`

### 11.3 TTS 做什么

TTS 子系统负责：

- 把中间汇报或最终结果播报给用户
- 配合 GUI 形成“我在做什么”的可感知体验

这次本地语音唤醒还额外复用了 TTS 做一段极短确认播报：“我在”。

### 11.4 常见落点

- 改 ASR provider 或接入逻辑：看 `voice/`
- 改本地唤醒模型、关键词生成或监听状态机：看 `voice/` 下的 wake-word 相关文件
- 改 TTS 生成与播放：看 `tts/`
- 改 GUI 中语音开关、唤醒词设置和交互方式：看 `gui/control_console.py`、`gui/floating/controller.py`、`gui/floating/controller_delegates.py`、`gui/floating/voice_controller.py`

## 12. 平台适配层

### 12.1 为什么单独有 `platform/`

因为桌面自动化高度依赖系统差异：

- 如何识别前台应用
- 如何打开应用
- 屏幕尺寸和缩放怎么处理
- 系统窗口如何激活
- Finder / 资源管理器怎么打开路径

这些都不应该散落到业务层，所以统一收敛在 `platform/`。

### 12.2 平台层的边界

平台层只处理“系统差异”，不负责业务编排。

简单说：

- “怎么在 macOS 打开 Finder” 是平台层问题
- “什么时候该打开目录” 是业务层问题

这个边界很重要。  
如果把业务逻辑塞进平台适配，很快会失控。

## 13. 配置体系

### 13.1 配置来源

配置由两部分合并：

- `src/baodou_ai/core/config.py` 中的 `DEFAULT_CONFIG`
- 仓库根目录本地 `config.json`

运行时会用本地配置覆盖默认值。

### 13.2 配置的作用

配置不仅控制 API key，还控制大量行为：

- 模型地址和 TLS 校验
- 截图后端
- 鼠标速度
- 最大迭代次数
- 过程汇报频率
- 后台 Agent provider
- 语音与 TTS 开关
- 悬浮球、伴随推荐、语言等 UI 行为

### 13.3 关键配置域

- `api_config`
  - 主模型地址、模型名、key、TLS
- `execution_config`
  - 最大步数、稳定检测、停滞策略、过程汇报
- `screenshot_config`
  - 截图 backend、fallback、调试截图
- `mouse_config`
  - 鼠标移动与安全参数
- `code_agent_config`
  - provider、并发、超时、命令模板
- `voice_interaction_config`
  - ASR、VAD、回声消除、空闲策略
- `wake_word_config`
  - 本地语音唤醒开关、唤醒词、阈值、冷却时间、模型目录
- `tts_config`
  - 播报模型与相关参数

### 13.4 新增配置项时必须同步改什么

这是非常高频、也非常容易漏的工作：

1. 改 `DEFAULT_CONFIG`
2. 改 `Config` 的读写或路径处理逻辑
3. 改 `control_console.py` 的设置页
4. 改 README 或开发文档中的配置说明
5. 补测试

如果只改第 1 步，往往会出现：

- 配置存在但 UI 不可编辑
- UI 有字段但运行时不读取
- 本地生效但文档不一致

### 13.5 本地语音唤醒相关配置要点

本地唤醒的配置入口是 `wake_word_config`，建议重点关注：

- `enabled`
  - 是否启用本地语音唤醒
- `phrases`
  - 唤醒词列表，当前默认是 `你好小彤` 与 `hey Lucy`
- `threshold`
  - 唤醒阈值
- `cooldown_ms`
  - 唤醒冷却，避免连续误触发
- `post_wake_timeout_seconds`
  - 唤醒后等待用户继续说话的超时时间
- `show_indicator`
  - 是否在悬浮球上显示唤醒状态提示
- `model_dir`
  - 本地模型目录，默认使用 `models/sherpa-onnx-kws-zipformer-zh-en-3M-2025-12-20`

实现上还需要注意一点：

- `Config.resolve_resource_path()` 会把相对路径解析成运行时绝对路径，因此模型目录可以安全地写成项目内相对路径

## 14. 错误处理与日志

### 14.1 结构化错误的目的

桌面自动化场景无法完全避免失败，但失败必须“可识别、可展示、可定位”。

因此项目对关键失败引入统一错误结构 `ErrorEnvelope`。

### 14.2 统一字段

- `source`
  - 错误来源，如截图、解析、工具、后台任务、语音、TTS
- `kind`
  - 错误类型，如 timeout、execution_failed、validation_failed
- `user_message`
  - 给用户看的说明
- `dev_detail`
  - 给开发者定位问题的细节
- `retryable`
  - 是否适合重试
- `code`
  - 稳定错误码
- `extra`
  - 补充上下文

### 14.3 日志系统做什么

运行时日志不是简单 `print` 收集，而是统一缓冲、统一转发。

这带来几个好处：

- GUI 日志窗口能稳定显示
- 结构化错误能被 UI 和测试识别
- 用户可见信息与开发调试信息可以分离

### 14.4 出错时优先排查顺序

建议按这个顺序看：

1. GUI 或控制台显示的用户提示
2. `RuntimeLogBuffer` 输出
3. `ErrorEnvelope` 内容
4. 对应子系统测试
5. 对应模块源码

## 15. 测试体系

### 15.1 测试是如何组织的

测试集中在 `tests/`，按模块横向铺开，而不是完全按目录镜像。

这意味着：

- 看测试文件名通常就能猜到模块边界
- 改某个功能时，应先搜对应 `test_*.py`

### 15.2 测试重点分布

- runner 与控制循环：`test_runner*.py`
- parser 与协议：`test_parser.py`
- 工具与自动化：`test_automation*.py`
- 截图与平台：`test_screenshot.py`、`test_platform_*`
- GUI：`test_floating_window.py`、`test_log_window.py`
- 后台任务：`test_code_agent_*`
- 语音交互与本地唤醒：`test_voice_interaction.py`、`test_wake_word_engine.py`
- 配置：`test_config.py`
- 模型下载脚本：`test_download_wake_word_model.py`

### 15.3 开发时怎么选测试

- 改协议或 parser：先跑 `test_parser.py`
- 改 runner 主流程：先跑 `test_runner*.py`
- 改工具执行：先跑 `test_automation*.py`
- 改截图：先跑 `test_screenshot.py`
- 改后台 Agent：先跑 `test_code_agent_*`
- 改语音/TTS/本地唤醒：先跑 `test_voice_interaction.py`、`test_wake_word_engine.py`
- 改设置或配置：先跑 `test_config.py`
- 改模型下载脚本：先跑 `test_download_wake_word_model.py`

### 15.4 验收脚本

仓库里还有 `scripts/` 下的验收脚本，用于 GUI、平台和能力分组验证。  
这部分适合在功能完成后做回归，而不是替代单元测试。

## 18. 新功能开发推荐流程

为了避免改动扩散失控，建议按下面方式做：

1. 先确认功能属于哪一层
   - 入口层
   - runner 主链路
   - 工具层
   - GUI 层
   - 平台层
   - 后台任务层
2. 找对应测试文件
3. 再改实现
4. 最后补文档和配置说明

一条经验：

- 先判断边界，再写代码。
- 这个项目最怕“明明是 A 层问题，却在 B 层硬修”。

## 21. 常见陷阱

### 21.1 误把入口当成业务核心

GUI/CLI/API 只是入口，绝大多数行为在统一内核里。

### 21.2 只改默认配置，不改设置页

结果就是：

- 代码里能读到
- UI 里改不了
- 文档也不一致

### 21.3 只接入本地唤醒代码，不处理模型目录和依赖

结果通常是：

- 运行时提示缺少 `sherpa-onnx` 或 `sentencepiece`
- 模型路径配置正确，但本地没有实际模型文件
- 开发机能跑，其他协作者拉代码后不能直接使用

因此本地语音唤醒改动需要把代码、配置、下载脚本、文档和 `.gitignore` 一起考虑

### 21.4 把平台逻辑写进业务层

短期看似方便，长期会让 macOS/Windows 分支越来越乱。

### 21.5 新增工具只改执行，不改 schema

模型可能根本不知道工具怎么用，或者参数校验直接失败。

### 21.6 只看 GUI 现象，不看 runner 和日志

很多表面问题其实是：

- prompt 输入有缺失
- parser 解析失败
- tool 执行失败
- 平台适配异常

## 22. 项目边界

当前项目明确不做：

- 用文档承诺尚未稳定的平台能力
- 让模型一次执行多步主动作
- 用重型监控系统替代当前轻量日志治理

这些边界能帮助你判断哪些方向是“应继续增强”，哪些方向是“应谨慎扩张”。

## 23. 一句话结论

这个项目的本质是：

一个以 `ControlLoopRunner` 为核心、以“单步观察与执行闭环”为基础、向 GUI、语音、后台 Code Agent 和跨平台能力扩展的桌面自动化系统。

如果你想快速上手开发，抓住四个词就够了：

- 统一入口
- 统一内核
- 清晰边界
- 先测后改
